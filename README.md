# tiny-db — своя SQL-база данных с нуля + веб-админка tinydb Studio

> **Это handoff-документ для следующего Клода.** Прочитай целиком перед работой.
> Пользователь общается по-русски, отвечай по-русски и **без буквы «ё»** (пиши «е»).

---

## Цель проекта

Построить **настоящий движок базы данных с нуля на TypeScript**, своими руками,
чтобы доказать инженерный навык. **Это не бизнес, а proof-of-skill.** Поверх
движка сделана графическая веб-админка **tinydb Studio** (как pgAdmin).

## Железные правила (не нарушать)

1. **Ядро БД пишем сами, без внешних библиотек.** Только `tsx` + `typescript` +
   `@types/node` (dev, для запуска и проверки типов). Никаких ORM, готовых БД,
   парсеров, editor-библиотек и т.п. — в этом весь смысл.
2. **Движок и его тесты не ломать.** Сейчас **314 тестов, все зелёные**. После
   любой правки: `npm run typecheck` и `npm test` должны быть чистыми.
3. **Идём по одной фиче/этапу за раз.** Собрал → написал тесты → запустил →
   показал результат → ждёшь команду пользователя. Не убегать вперёд.
4. Маленькие чистые файлы. После фичи коротко объясни, что сделал и где граница.

## Стек

TypeScript (strict), Node.js 20+, ESM (`"type":"module"`), запуск через `tsx`
(без сборки). Свой мини-раннер тестов (без jest/vitest). Импорты с расширением
`.ts` (в tsconfig `allowImportingTsExtensions`).

## Как запускать

```bash
cd "/Users/agent/Desktop/BD Subd/tiny-db"
npm install            # один раз

npm test               # 314 тестов
npm run typecheck      # проверка типов

npm run repl mydata    # SQL-консоль в терминале (папка данных mydata)
npm run repl:kv data.db# старая KV-оболочка (этапы 1-2)

npm run studio studio-data 8137   # веб-админка: http://localhost:8137
                                  # логин/пароль admin/admin (env STUDIO_USER/STUDIO_PASS)
```

**ВАЖНО про порт:** 8080 занят чужим Open WebUI на машине пользователя — используй
**8137** (или другой свободный). Формат: `npm run studio <корень-баз> <порт>`.
Останавливать: `lsof -ti tcp:8137 | xargs kill`.

**Безопасность (по умолчанию):** сервер слушает **только loopback `127.0.0.1`** —
из локальной сети не виден. Открыть наружу можно осознанно через
`STUDIO_HOST=0.0.0.0`, но тогда обязательно смените логин/пароль
(`STUDIO_USER`/`STUDIO_PASS`) — иначе любой в сети войдёт по admin/admin. Вход:
constant-time сравнение, анти-брутфорс (10 неудач с адреса -> блок на минуту),
сессии живут 8 часов. Ответы: `X-Content-Type-Options: nosniff`; страница отдаётся
с CSP (`default-src 'none'`, inline-скрипт/стиль, `img-src data:`, `connect-src 'self'`)
и `X-Frame-Options: DENY`. Подробнее — в блоке про безопасность ниже.

**ВАЖНО про сервер:** держи его запущенным, пока пользователь работает. Раньше я
останавливал сервер после каждой проверки — пользователь потом видел «Failed to
fetch». Если он жалуется на эту ошибку — просто подними сервер снова.

Инспекция данных на диске: `npm run inspect:log <файл>` (лог), `npm run inspect:tree <файл>.tbl` (B+-дерево).

## Что построено — ДВИЖОК (все 10 этапов плана + доработка)

1. **Хранение** — append-only лог + индекс в памяти + replay, durability (fsync
   до обновления памяти; тест через реальный отдельный процесс), устойчивость к
   обрыву записи.
2. **Компакция лога** — атомарная замена через temp-файл + fsync + rename.
3. **Страницы + B+-дерево** — 4КБ-страницы, дерево на диске, O(log n), split,
   упорядоченный обход по связанным листьям.
4. **Таблицы/схема/типы** — CREATE TABLE (INTEGER/TEXT), сериализация строки,
   order-preserving ключи (INTEGER сортируется численно, включая отрицательные),
   durable каталог схем (в лог-движке).
5. **Парсер SQL** — лексер + рекурсивный спуск -> AST.
6. **Исполнение** — CREATE/INSERT/SELECT реально работают.
7. **Вторичные индексы** — B-дерево на колонку, планировщик (WHERE col=X по
   PK/индексу vs скан; в SELECT есть scanned/usedIndex).
8. **Транзакции + WAL** — BEGIN/COMMIT/ROLLBACK, redo-восстановление после сбоя
   (persist->checkpoint), атомарность по нескольким файлам (данные+индексы).
9. **Конкурентность** — strict 2PL, блокировки таблиц S/X, no-wait (без deadlock),
   несколько одновременных транзакций (db.beginTransaction()).
10. **Джойны/агрегаты/планировщик** — INNER JOIN (эквиджойн, алиасы, t.col),
    COUNT/SUM/MIN/MAX/AVG, GROUP BY, index nested loop join.

**Доработка сверх плана:** UPDATE и DELETE (полный CRUD в SQL). Плюс
`Database.dropTable()` — удаление таблицы с индексами и файлами, durable
(tombstone в каталоге), имя после этого свободно. В SQL команды DROP пока нет —
метод дёргает кнопка «удалить таблицу» в Studio (эндпоинт `POST /api/tables/delete`).

**SQL-подмножество:** CREATE TABLE, CREATE INDEX, INSERT, SELECT (* / колонки /
агрегаты, JOIN ... ON, WHERE с AND/OR/скобками и операторами =,!=,<>,<,>,<=,>=,
GROUP BY), UPDATE ... SET ... WHERE, DELETE ... WHERE, BEGIN/COMMIT/ROLLBACK.
Нет: ORDER BY, LIMIT, LEFT JOIN, DROP, NULL, UNIQUE, IN/BETWEEN.

## Что построено — tinydb Studio (веб-админка)

Сервер `src/server.ts` (node:http, без зависимостей), страница `web/tinydb-studio.html`
(голый HTML/CSS/JS). Движок используется как есть, не переписан.

**Маршруты:** `GET /` (страница), `POST /api/login`, `POST /api/logout`,
`GET /api/databases`, `POST /api/databases`, `POST /api/databases/delete`,
`GET /api/tables?db=`, `POST /api/tables/delete {db,table}`,
`POST /api/query {db,sql,noHistory?}`, `POST /api/insert`, `POST /api/update`,
`POST /api/delete`, `GET /api/history`, `POST /api/history/clear`.
Все `/api` (кроме login) требуют cookie-сессию.

**Реализованные фичи (каждая делалась отдельно, по команде пользователя):**
- **Авторизация** — логин/пароль (admin/admin по умолчанию, env STUDIO_USER/PASS),
  сессия в HttpOnly-cookie, экран входа.
- **Несколько баз** — каждая база = своя директория движка внутри корня; создание
  базы из UI. Служебные базы с префиксом `_` скрыты (см. историю ниже).
- **Полноэкранный вид**, брендинг, кнопка «Выйти».
- **Вкладки «Данные» / «Схема»** — Схема показывает таблицы, колонки, типы, PK
  (только чтение; берёт из расширенного /api/tables с полем `schema`).
- **Редактирование ячейки** — клик по ячейке -> инпут, Enter сохранить, Esc отмена;
  POST /api/update.
- **Добавление строки** — кнопка «+ строка» -> форма по колонкам; POST /api/insert.
- **Удаление строки** — иконка корзины -> **своё** инлайновое подтверждение
  (да/нет, НЕ нативный confirm); POST /api/delete.
- **Экспорт результата** — кнопка «Экспорт» -> CSV или JSON, целиком в браузере
  (Blob), CSV с экранированием по RFC4180 + BOM для Excel. Сервер не трогается.
- **История запросов** — успешный /api/query пишется в служебную таблицу
  `query_history(id,sql,created_at)` в **отдельной базе `__studio`** (через движок,
  durable). Список в сайдбаре, клик -> запрос обратно в редактор. Панель:
  свой скролл (max-height), схлопывание подряд идущих дублей (`×N`), последние 15
  + «показать все», иконка-корзина «очистить» (`POST /api/history/clear`).
  **В историю пишутся только ручные запросы** (кнопка «Выполнить»); авто-запросы
  (клик по таблице, импорт) идут с флагом `noHistory:true` и не засоряют историю.
- **Подсветка синтаксиса** — прозрачная textarea поверх подсвеченного `<pre>`,
  свой токенайзер (ключевые слова/строки/числа/комментарии). Логика выполнения
  не тронута (runSql читает sql.value).
- **Удаление базы** — кнопка «🗑 Удалить БД» справа вверху (инлайн-подтверждение),
  `POST /api/databases/delete` (закрыть базу + стереть директорию; `__studio` защищена).
- **Удаление таблицы** — иконка-корзина у таблицы в сайдбаре (инлайн-подтверждение,
  как в истории), `POST /api/tables/delete` -> `Database.dropTable()`. После можно
  создать таблицу с тем же именем. Ручной CREATE/DROP через «Выполнить» обновляет
  список таблиц слева.
- **Импорт SQL** — кнопка «↑ Импорт SQL» рядом с «Выполнить»: выбор `.sql`-файла,
  чтение на клиенте (FileReader) и выполнение скрипта (несколько инструкций) против
  текущей базы; в историю не пишется.
- **Смена языка RU/EN** — кнопка-переключатель в шапке; словарь `I18N` +
  `data-i18n`/`data-i18n-ph`/`data-i18n-title` в разметке + `tr()` для динамики;
  выбор запоминается в `localStorage`. Тексты ошибок с сервера (из движка) —
  данные, остаются на русском.
- **Favicon** — SVG-логотип (синее кольцо ◉) как data-URI в `<head>`.

Редактирование/добавление/удаление доступно только когда результат — **одна
таблица с PK среди колонок** (иначе строку не идентифицировать); JOIN/агрегаты =
только чтение. Это определяется на клиенте (`computeEditContext`).

### Безопасность (после аудита)

Модель угроз: локальный однопользовательский инструмент. Сделано:
- **Только loopback** — `server.listen(port, host)`, host = `STUDIO_HOST` или
  `127.0.0.1`. Раньше слушал `0.0.0.0` (вся сеть) — это была главная дыра.
- **Вход:** сравнение логина/пароля за постоянное время (`crypto.timingSafeEqual`,
  без short-circuit) — нет timing-атаки; анти-брутфорс по адресу
  (`LOGIN_MAX_FAILS`=10 -> блок `LOGIN_LOCK_MS`=60с, ответ 429).
- **Сессии** живут `SESSION_TTL_MS`=8ч (было — вечно). Cookie: `HttpOnly`,
  `SameSite=Strict` (защита от CSRF), путь `/`.
- **Заголовки:** все ответы — `X-Content-Type-Options: nosniff`; страница —
  CSP (`default-src 'none'`; inline script/style; `img-src data:`; `connect-src 'self'`;
  `base-uri/form-action 'none'`; `frame-ancestors 'none'`) и `X-Frame-Options: DENY`.
- **Тело запроса** ограничено 1 МБ с корректным reject (раньше промис зависал).
- **`esc()`** на клиенте экранирует и кавычки (defense-in-depth для атрибутов).

Уже было надёжно (не трогал): идентификаторы SQL — строго `[A-Za-z_][A-Za-z0-9_]*`
(лексер), поэтому инъекции и **path traversal** через имена таблиц/колонок/баз
невозможны; значения в INSERT/UPDATE/DELETE идут через `formatLiteral` (экранирование
кавычек / проверка INTEGER); имена баз — `NAME_RE`; `deleteDatabase`/`dropTable`
удаляют файлы только по ключу из реестра открытых баз; служебная `__studio` не в
реестре — снаружи не запросить; статика — только фиксированный html, без обхода пути.

Осталось на усмотрение (не критично для loopback): дефолт admin/admin (при старте —
предупреждение в консоль; для `0.0.0.0` смена обязательна), нет HTTPS (localhost),
`loginFails`/`sessions` без фонового GC (протухшие чистятся лениво при обращении).

## Заметки по рабочему процессу (важно!)

- **Проверяй в браузере через claude-in-chrome MCP.** НО: клики по координатам
  «плавают» (скриншот масштабируется, размеры не совпадают с вьюпортом). Логин
  делай через `find` + клик по `ref`; мелкие клики/подтверждения удобнее
  прогонять через `javascript_tool` (вызывать те же функции страницы: `showApp()`,
  `buildAddForm()`, `askDelete()`/`doDelete()`, `setView('schema')`, `setSql()`
  и т.д.). Это легитимная проверка — обработчики привязаны и работают.
- **Не запускай реальные скачивания файлов** в браузере при проверке экспорта —
  проверяй сгенерированный CSV/JSON строкой через `javascript_tool`, чтобы не
  сорить в Загрузках пользователя.
- Порт 8080 занят (Open WebUI) — всегда бери свободный (8137).
- Каждую серверную фичу параллельно проверяй ещё и через `curl` (логин ставит
  cookie в `-c cookie.txt`, дальше `-b cookie.txt`).
- Временные данные — в `/tmp/...`, чистить за собой. Для «живого» сервера
  пользователю используй постоянную папку `studio-data` (внутри проекта).

## Идеи на будущее (если попросят «дорабатывай движок»)

ORDER BY / LIMIT, LEFT JOIN, DROP TABLE/INDEX **в SQL** (движковый
`Database.dropTable()` уже есть — осталось добавить парсер/executor; DROP INDEX
ещё нет вовсе), WHERE IN/BETWEEN, NULL, UNIQUE, row-level locking или MVCC,
многопроцессный доступ. Строго по одной фиче.
